# Compounding 루프 운영 룰

verifier가 제안한 개선 후보를 `compounding-applied.md`로 역류시킬 때의 판단 원칙.
도메인 지식(HR 원칙)이 아닌 **Claude 세션의 운영 가드**.

## 핵심 원칙

### 1. 1건 관찰은 보류, 2회+ 반복 패턴만 승격
- 단일 시나리오에서만 보인 권고는 **하네스(플레이북·템플릿·원칙) 수정 금지**
- 동일 패턴이 **2회 이상** 독립 실행에서 재발하면 그때 승격
- 보류 이유를 `compounding-applied.md`에 명시 → 다음 회차 판정 근거로 활용

### 2. 자기 교정은 정상 작동 신호
- "내가 이전 회차에 도입한 기준이 다음 회차에서 실패함" → 실패가 아니라 하네스의 정상 작동
- 기준을 없애지 말고 **구조로 재정의** (예: 라인 수 → 필수 구성요소)

### 3. 역류 대상 층위 구분
- **지식 수준 (knowledge/)**: HR 제도 설계 원칙·안티패턴
- **절차 수준 (playbooks/)**: 실행 단계·체크리스트
- **형식 수준 (templates/)**: 산출물 구조·섹션 구성
- **세션 수준 (.claude/rules/)**: Claude가 지킬 운영 가드 (이 파일)
- 한 권고를 복수 층에 중복 반영하지 않음. 가장 적합한 한 곳에만.

### 4. 관찰 보류 재평가 시점
- 관찰 보류 건은 **다음 2회차 실행 완료 후** 재평가
- 재발 시: 본질 적용 또는 룰 승격
- 재발 안 함: 원 시나리오 국소 특이 케이스로 분류, 기록만 남김

## `compounding-applied.md` 필수 구조

1. **적용 (N)**: 변경 파일·변경 전/후·근거·기대 효과
2. **보류 (M)**: 보류 이유 (근거 부족 / 단건 관찰 / 국소 특이)
3. **누적 변화표**: 회차 × 파일 매트릭스 (전 회차 비교)
4. **다음 회차 예상 검증 포인트**: 이번 변경이 다음 회차에서 PASS 되는지 체크할 기준

## Verifier 호출 원칙

- Verifier는 **독립 컨텍스트**로 호출 (원본 실행 세션과 분리)
- 산출물 수정 금지, **판정만**
- `scenarios/verifier-prompt.md` 템플릿 엄수

---

*이 룰은 compounding 루프가 "개선 같지만 실은 과적합"이 되지 않도록 하는 보호 장치.*
*새 패턴이 이 룰 자체를 수정해야 할 정도면 2회 이상 독립 재발을 확인한 뒤 갱신.*
