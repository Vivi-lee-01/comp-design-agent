# Scope 판정 규율 — 모든 평가보상 설계 플레이북 공통

본 문서는 `design-pay-band`, `design-leveling`, `design-review-cycle` 등 **모든 평가보상 설계 플레이북**에 공통으로 적용되는 **Scope 판정 절차**를 정의한다. 플레이북 본문에 중복 서술하지 않고 여기 한 곳에서 관리하기 위한 세션 운영 층 룰이다.

## 1. 근거 (승격 히스토리)

- **4회차** (design-pay-band × 4): s3에서 "Total Comp 층 vs Cash 층 분리 선언" 자발 발화 → `playbooks/design-pay-band.md` §2-보강으로 승격 (pay-band 국소)
- **5회차** (design-leveling × 4): 4개 시나리오 모두 §0 Scope 판정 섹션 자발 발화 → `playbooks/design-leveling.md` Step 1-보강으로 승격 (leveling 국소, pay-band 패턴 재현)
- **6회차** (design-review-cycle × 4): 4/4 시나리오 §0 "본 산출물의 Scope 판정" 자발 발화, 3 플레이북 × 4 시나리오 누적 **9/12 = 75%** 자발 발화 관찰
- **7회차** (design-leveling × 4 재실행): Rules Read 4/4, §0 5단계 (a)-match 4/4, Rollback 3신호 0/12. A verdict (진짜 작동) → rules 층 안정성 재확인
- **8회차** (design-diagnose × 4): 4/4 시나리오 §0 (a)-match, 이관 4열 완결률 **100% (20/20 셀)**, Rollback 3신호 0/12 감지. **4번째 플레이북 재현**으로 "3 플레이북 국소"에서 **하네스 공통 원칙**으로 최종 확정.

`.claude/rules/compounding.md` §1("2회+ 반복 패턴 승격") 조건을 **세 번째 플레이북에서도 동일 본질 패턴이 재현**했으므로 도메인 국소가 아닌 **하네스 공통 원칙**으로 판정, 세션 운영 층으로 승격. 8회차 diagnose 재현으로 확정.

## 2. §0 Scope 판정 5단계 절차

모든 평가보상 설계 플레이북 실행 시, executor는 **output 첫 섹션(§0)**에서 다음 5단계를 수행한다.

1. **Expected 정독**: `expected.md` 전체를 먼저 정독하고, 그 시나리오가 실제로 요구하는 것과 플레이북이 제공하는 전체 단계를 **대조**한다.
2. **단계 4분류**: 각 플레이북 단계/본문 항목을 **본격 / 축소 / 이관 / 생략** 네 범주 중 하나로 분류한다.
3. **근거 인용**: 분류 근거를 `expected.md` 구체 인용으로 명시한다(예: "P1 #5", "난이도 태그 Easy", "다뤄질 필요 없는 사항" 문장). 주관 판단("~같아 보임")은 근거가 아니다.
4. **이관 인터페이스 명시**: "이관" 분류 항목은 **이관 대상 Task + 인터페이스 계약(입력/출력)**을 함께 적는다. 흐릿한 이관("~영역에서 다룸")은 금지.
5. **생략 사유**: scope 밖으로 밀어낸 항목도 "왜 이번에는 안 하는가"를 한 줄로 남긴다(재개 시점·담당 불명이면 "미정").

## 3. 4분류 정의

| 분류 | 의미 | 필수 조건 |
|------|------|-----------|
| **본격** | 플레이북 단계 표준 절차를 완전 실행 | — |
| **축소** | 구조는 유지하되 분량·산식·세부를 경량화 | 경량화 근거를 expected.md 인용으로 명시 |
| **이관** | 본 산출물 책임이 아니며 다른 task로 밀어냄 | 이관 대상 task 이름 + 인터페이스 계약(입/출력) 동반 |
| **생략** | expected.md가 해당 레이어 자체를 요구하지 않음 | "다뤄질 필요 없음" 근거 인용 또는 scope-out 사유 한 줄 |

## 4. 자가 체크 3문항 (output 말미 또는 체크리스트에 포함)

- [ ] §0 Scope 판정 섹션이 output 앞부분(서문 또는 §0)에 존재하는가
- [ ] "이관" 항목은 4열 표(항목 · 대상 task · 사유 · 인터페이스 입/출력)로 표기했는가
- [ ] "축소"·"생략" 판정은 `expected.md` 인용 근거를 달았는가

위 3문항 중 하나라도 No면 executor는 해당 섹션을 보완한 뒤 output을 마감한다.

## 5. 안티패턴 (금지)

1. **보수 편향**: "플레이북 전체 단계를 기계적으로 완주하면 안전하다"는 태도. → 과잉 스펙을 생산하고 시나리오와 동떨어진 산출물을 만든다.
2. **Overshoot**: expected.md 명시 범위 밖까지 "혹시 몰라서" 포함하는 습관. → 축소/생략 판정이 약한 executor의 전형적 실패.
3. **§0 누락**: scope 판정 없이 바로 본격 단계(Step 1, Step 2…)로 진입. → 이 경우 output은 "선언 없는 암묵 축소"가 되어 verifier가 판정 근거를 잡을 수 없음.
4. **주관 근거**: "이 조직에는 ~가 맞아 보임" 같은 주관 판단을 근거로 사용. → 반드시 `expected.md` 또는 `context/` 문서를 인용.
5. **이관 흐리기**: "페이밴드 영역에서 다룸"처럼 대상 task와 인터페이스를 생략한 채 이관 선언. → 이관은 4열 표(항목·대상·사유·인터페이스)로만 인정.

## 6. 플레이북 본문과의 관계

각 플레이북 본문에는 "Scope 판정" 구체 절차를 **중복 서술하지 않는다**. 다음 한 줄만 삽입하여 본 규율을 참조한다.

> Step N-보강 (Scope 판정): 본 플레이북 진입 전 `.claude/rules/scope-discipline.md`의 §0 Scope 판정 5단계를 수행하라. [플레이북 고유 예시 한 줄]

플레이북별 고유 예시:
- **design-pay-band**: "Total Comp 층과 Cash 층 분리는 **축소/이관** 판정의 대표 예시다."
- **design-leveling**: "'레벨 구조는 유지하되 재보정'형 시나리오에서는 **축소** 판정이 기본값이다."
- **design-review-cycle**: "Review cycle은 peer feedback·calibration·compensation link·career dialogue 레이어가 공존하므로 특히 **이관 인터페이스(→ pay-band, → leveling)** 명시가 필수다."
- **design-diagnose**: "진단은 전체 제도 스캔이지만 개별 이슈의 해결은 pay-band/leveling/review-cycle로 **이관**되므로 **이관 인터페이스(→ 대상 task · 입/출력 계약)** 명시가 특히 중요한 플레이북이다."

## 7. 롤백 조건 (리스크 신호 감지 시)

rules 층 승격은 되돌리기 어려우므로, 다음 신호가 관찰되면 재검토한다.

- **과잉 작동**: 특정 도메인 템플릿 용어(예: pay-band의 "Min/Mid/Max", "n<50 표본")가 본 scope 판정 절차를 통해 **다른 도메인 산출물에 침투**
- **형식화**: §0이 형식만 채우고 실질적 분류 근거가 빈약한 출력이 다수 관찰
- **생략 남용**: "이관/생략"이 근거 없이 남발되어 산출물이 expected.md 주요 이슈를 놓침

이 중 2건 이상 2회차 연속 관찰 시 본 규율을 부분 개정 또는 조건부 작동으로 재설계.
