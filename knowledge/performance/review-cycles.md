# Performance Review Cycles (평가 사이클)

## 개념

**평가 사이클**은 정기적으로 구성원의 성과/역량을 평가하고 피드백하며, 보상/승급 의사결정으로 이어지는 전체 프로세스.

## 사이클 주기

### 연 1회 (Annual)
- 장점: 간결, 운영 부담 적음, 장기 관점
- 단점: 피드백 주기 길어 실시간 교정 어려움
- 적합: 중견/대기업, 안정적 조직

### 연 2회 (Semi-annual)
- 장점: 중간 교정 가능, 신입 온보딩 사이클 반영
- 단점: 관리 부담 2배
- 적합: 성장 중인 스타트업/스케일업

### 분기 (Quarterly)
- 장점: 빠른 피드백, OKR과 연동 쉬움
- 단점: 매우 큰 관리 부담, 피로도
- 적합: 초기 스타트업, 급성장 조직 (단, 간소화 필수)

### Continuous (상시)
- 공식 사이클 없음. 상시 1on1 + 프로젝트별 리뷰
- 장점: 실제 업무에 밀착
- 단점: 일관성 부족, 보상 의사결정 어려움
- 적합: 크리에이티브/R&D 조직

## 표준 사이클 구성 요소

### 1. Goal Setting (목표 설정)
- 사이클 시작 시
- 회사/팀 목표와 연결
- 상사-부하 합의

### 2. Mid-cycle Check-in (중간 점검)
- 사이클 중간
- 진행 상황 점검, 방해 요인 제거
- 목표 조정 가능

### 3. Self Review (자기평가)
- 사이클 말 진입 시
- 본인의 성과/역량 평가
- 사례 중심 기술 권장

### 4. Peer Review (동료평가)
- 선택적
- 프로젝트 협업자 2~5명
- 360 리뷰의 일부

### 5. Manager Review (매니저 평가)
- 부하의 성과 종합 평가
- 근거와 사례 필수
- 차기 사이클 개발 계획 포함

### 6. Calibration (조율)
- 매니저들이 모여 평가 결과 조율
- 평가 인플레이션/편향 방지
- **중요**: 비공개가 아님, 조정 기준 명확해야

### 7. 평가 면담 (Performance Conversation)
- 매니저-부하 1:1
- 피드백 전달 및 개발 계획 협의
- 보상 결정 **전** 또는 **후** 명확히 구분

### 8. 보상 결정 (Compensation Decision)
- 평가 결과를 바탕으로 인상/보너스 결정
- 페이밴드와 연동
- People/HR이 조정

### 9. 커뮤니케이션 (Communication)
- 최종 결과 전달
- 문서화
- 후속 개발 플랜 실행

## 평가 축 설계

### 성과 (What) 축
**달성한 결과**. OKR 달성도, KPI 등.

### 역량/행동 (How) 축
**어떻게 일했는가**. 조직 가치 실천, 레벨별 역량 충족.

**두 축 분리가 중요**: 
- 결과만 보면 **수단을 가리지 않는 행동** 용인
- 과정만 보면 **성과 없이 열심히만** 하는 사람 과대평가

### 잠재력 (Potential) 축 (선택)
**차기 레벨 가능성**. 승급 결정에 활용.

## 평가 등급 설계

### 5단계 (권장)
- Far Exceeds Expectations
- Exceeds Expectations
- Meets Expectations (대부분)
- Below Expectations
- Far Below Expectations

### 3단계 (간소화)
- Exceeding / Performing / Developing

### 2단계 (최소)
- Strong / Needs Support

**원칙**:
- 등급 수 적을수록 관리 용이
- 너무 적으면 차등 불가 (모두 "Performing")
- **등급별 분포 가이드** 제시 권장 (강제 아님)

## Calibration 프로세스

평가의 일관성을 위한 **매니저 간 조율**.

### 목적
- 매니저별 평가 관대/엄격 차이 보정
- 전사 차원 등급 분포 확인
- 승급 후보 검증

### 방식
1. 매니저가 본인 팀 잠정 평가 제출
2. 매니저 그룹(팀장급 또는 부서 단위)이 모여 검토
3. 특정 구성원에 대해 **근거 제시 및 조정**
4. 합의된 결과 확정

### 주의
- **순위 매기기(ranking)가 아님**. 등급 분포 확인이 목적.
- 근거 없는 하향 조정은 매니저-부하 신뢰 훼손.
- Calibration 내용은 **기록으로 남김**. 나중에 추적 가능하도록.

## Forced Distribution 논란

**Forced Distribution** (강제 배분): 상위 20% / 중위 70% / 하위 10% 등 비율 강제.

### 장점
- 평가 인플레이션 방지
- 저성과자 식별 강제

### 단점
- 팀 내 협력 저해 (zero-sum)
- 우수 인재로만 구성된 팀에서 **강제로 하위 등급** 부여 → 이직 유발
- GE가 유명하게 도입했다가 **폐지** (2015)

### 권장
- **강제 아닌 가이드 분포** 제시: "대부분의 팀에서 이 분포가 나타날 것으로 기대한다"
- 크게 벗어나면 **설명 요구**, 자동 조정 아님

## 평가 ↔ 보상 연결

### 직접 연결 (Tight Coupling)
- 평가 등급 → 인상률 매트릭스
- 투명, 예측 가능
- 단점: 평가가 "돈 결정 도구"가 되어 대화 왜곡

### 느슨한 연결 (Loose Coupling)
- 평가는 성장 대화 중심
- 보상은 별도 절차 (예산/시장 고려)
- 단점: 평가 영향력 희석

### 분리 권장 (현대적 접근)
- 평가 면담 = 성장/피드백 (봄)
- 보상 면담 = 결정 통보 (가을)
- 사이에 **경영진/HR 조정** 단계

## 연간 사이클 예시 (반기 기준)

```
1월  : H1 목표 설정
2~4월: 1on1, 중간 체크인
5월  : Self-review 작성
6월  : Peer review + Manager review
6월 말: Calibration, 보상 결정
7월  : 면담 + 인상 반영
7월  : H2 목표 설정
...반복
```

## 흔한 실수

### 1. Annual Review만 하고 1on1 없음
연말에 한 번에 피드백 = **서프라이즈 평가**. 상시 1on1 필수.

### 2. 평가 기준 불명확
"잘했다"의 기준이 사람마다 다름. **레벨별 기대치 문서화** 필수.

### 3. Calibration 스킵
매니저 간 기준 다름. 조율 안 하면 불공정.

### 4. 평가 면담 = 보상 통보
평가 면담에서 바로 "올해 인상률은..." → 대화가 돈 중심으로 왜곡.

### 5. 개발 계획 부재
평가만 하고 "다음에 뭘 할지"가 없음. 성장 도구로서 실패.
