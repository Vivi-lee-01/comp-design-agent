# Leveling (직무 레벨링)

## 개념

레벨링은 **직무의 성숙도/책임/영향력을 단계별로 정의**하는 체계. 페이밴드의 전제이며, 승급/승진 기준의 기반.

## 왜 필요한가

- 페이밴드를 만들려면 "누가 어디 속하는지" 정의가 먼저 필요
- 성장 경로(career path)가 명확해야 리텐션 상승
- 평가 기준이 레벨별로 다르기 때문 (IC3에게 요구되는 것과 IC5에게 요구되는 것은 다름)

## 기본 구조: Dual Track

대부분의 현대 조직은 **IC(Individual Contributor)**와 **Manager** 트랙을 분리.

```
IC Track:        Manager Track:
IC1 (Junior)     —
IC2              —
IC3 (Mid)        M1 (Team Lead)
IC4 (Senior)     M2 (Manager)
IC5 (Staff)      M3 (Senior Manager)
IC6 (Principal)  M4 (Director)
IC7 (Distinguished) M5 (VP)
```

**핵심 원칙**:
- 같은 숫자 레벨은 **동등한 보상 범위**
- IC→M 이동은 **승급이 아닌 방향 전환** (demotion이 아님)
- 매니저가 되지 않아도 시니어 이상 성장 가능해야 함

## 레벨 정의 축 (4대 축)

각 레벨을 정의할 때 아래 4가지를 명시:

### 1. Scope of Impact (영향 범위)
- IC1: Task 단위 작업
- IC2: Feature 단위 기여
- IC3: Team 단위 영향
- IC4: 여러 Team 영향
- IC5: Org 전체 영향
- IC6+: Company/Industry 영향

### 2. Autonomy (자율성)
- IC1: 상세 지시 필요
- IC2: 가이드라인 하에 자율
- IC3: 목표 주면 방법 자율
- IC4: 목표까지 스스로 정의
- IC5+: 조직 방향 정의에 참여

### 3. Complexity (복잡도)
- IC1: 잘 정의된 문제
- IC2: 일부 모호한 문제
- IC3: 모호한 문제, 명확화 필요
- IC4: 다영역 걸친 복합 문제
- IC5+: 문제 자체를 발견/정의

### 4. Influence (영향력)
- IC1: 본인 작업 완수
- IC2: 팀 내 기술적 기여
- IC3: 팀 내 리더십 (기술/프로세스)
- IC4: 타 팀과 협업 주도
- IC5+: 조직 문화/방향 형성

## 직군별 차이

Job Family마다 세부 기준이 다름.

**Engineering**: Scope/Complexity 비중 크다.
**Product**: Influence/Ambiguity 비중 크다.
**Design**: Craft depth와 Systems thinking이 핵심.
**Sales**: Quota/Deal size가 레벨과 연동되기도 함.
**Operations**: Process ownership 범위가 축.

## 설계 단계

### Step 1: Job Family 정의

현재 존재하는 직무들을 grouping. 너무 잘게 쪼개지 말 것 (5~10개가 현실적).

### Step 2: 레벨 수 결정

조직 규모에 따라:
- **< 30명**: 3~4 레벨 (Junior / Mid / Senior / Lead)
- **30~100명**: 4~5 레벨
- **100~500명**: 5~7 레벨
- **500명+**: 7~10 레벨

너무 많으면 **관리 불가**, 너무 적으면 **성장 경로 부족**.

### Step 3: 레벨별 기대치 문서화

각 레벨에 대해 4대 축으로 기대치 기술. 가능하면 **구체적 예시**와 함께.

**예시 (Engineering IC3)**:
> - Scope: 한 팀의 기능을 end-to-end로 구현
> - Autonomy: 요구사항 받으면 설계부터 배포까지 자율 수행
> - Complexity: 기존 시스템과의 통합 설계 가능
> - Influence: 팀 내 코드 리뷰 주도, 주니어 멘토링
> - 구체 예: "결제 모듈 리팩토링을 혼자 설계하고 2주 내 배포"

### Step 4: 승급 기준 정의

IC3 → IC4가 되려면 무엇이 충족되어야 하는가?

- **최소 재직 기간**: 현 레벨에서 최소 N개월
- **지속적 성과**: 최근 N회 평가에서 기대치 이상
- **다음 레벨 역량 입증**: "이미 IC4처럼 일하고 있다"는 증거
- **승급 심사**: 매니저 추천 + 패널 리뷰 or 칼리브레이션

### Step 5: 현직자 매핑

기존 구성원을 새 레벨 체계에 매핑.

**주의사항**:
- 매핑은 **보상에 영향** (기존 직급/연봉과 충돌 가능)
- 다운레벨링은 **매우 조심스럽게**. 기존 보상은 유지하되 레벨만 조정하는 "red circle" 방식 있음
- 매핑 결과를 본인에게 **설명 가능**해야 함

## 흔한 실수

### 1. 연차 기반 레벨링
"10년차니까 IC5" 식의 연차 기반은 실제 역량과 괴리. **역량 기반**이어야 함.

### 2. 직급명 혼동
"차장/부장" 같은 전통 직급과 IC/M 레벨을 섞으면 혼란. **하나의 체계로 통일**.

### 3. 레벨 수 과다
스타트업이 7레벨 만들면 대부분의 레벨이 비어있거나 기준이 모호해짐.

### 4. 승급 기준 모호
"리더십을 보여라"는 기준이 아님. **관찰 가능한 행동**으로 기술.

### 5. IC/M 비대칭
IC 트랙이 IC4까지만 있고 M 트랙이 M6까지 있으면 **은연중 매니저 트랙 우대** 신호.
