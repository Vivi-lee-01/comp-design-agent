# Evaluation Rubric

플레이북 산출물을 평가하는 공통 체크리스트. verifier 에이전트가 **독립 컨텍스트**에서 이 기준으로 Pass/Fail을 매긴다.

> **Knowledge 의존성 (SSOT)**
>
> 이 rubric의 일부 항목은 외부 지식·플레이북 문서의 **현재 등재 내용**에 의존한다. 지식/플레이북 문서가 확장·축소될 때는 아래 표의 **현 수치**를 먼저 갱신한 뒤 해당 rubric 항목을 동기화한다. 스캔 절차는 `.claude/rules/knowledge-sync.md` 참조.
>
> | 의존 대상 | 원천 문서 | 현 수치 | 참조 rubric |
> |-----------|-----------|---------|-------------|
> | 원칙 목록 | `knowledge/principles.md` §1~§7 | 7개 | D-1 |
> | 안티패턴 목록 | `knowledge/principles.md` §안티패턴 | 6개 | D-2 |
> | 구조적 누락 체크리스트 | `playbooks/diagnose.md` Step 4 | 8개 | D-3 |

## 채점 방식

- **PASS** — 항목 조건이 산출물에서 명시적으로 충족
- **WEAK** — 흔적은 있지만 품질 부족 (예: 표는 있는데 값 비어있음)
- **FAIL** — 조건 미충족 또는 완전히 누락
- **N/A** — 해당 플레이북/시나리오에서 적용 불가

집계:
- 공통 L1 항목 전부 PASS → 기본 품질 통과
- 플레이북별 L1 항목 전부 PASS → 해당 플레이북 품질 통과
- PASS 합계 ≥ 80% 이면 "루프 통과"

## 공통 (모든 플레이북 공통) [L1 5개, L2 3개]

| ID | L | 체크 항목 | 판정 방법 |
|----|---|-----------|-----------|
| G1 | L1 | 모든 주장에 **근거** (문서·데이터·관찰)가 붙어 있다 | 산출물에서 임의로 5개 주장 샘플링 → 근거 인용 확인 |
| G2 | L1 | **추측과 사실이 구분**되어 있다 | "추정컨대", "가정", "사실" 같은 메타 언어 또는 별도 섹션 존재 |
| G3 | L1 | **민감 정보 노출 없음** — 개인명/연봉 원값/이메일 등 식별자 미포함 | 원본 데이터 있어도 익명 코드(A1, A2)로 치환 |
| G4 | L1 | **제약 (`constraints.md`) 위반하는 권고 없음** | 산출물 권고 각각을 constraints와 대조 |
| G5 | L1 | 산출물이 **템플릿 구조**(`templates/*`)를 따른다 | 필수 섹션 누락 없음 |
| G6 | L2 | **우선순위/로드맵**이 포함되어 있다 | 단일 리스트가 아닌 Impact/Urgency/Feasibility 같은 2+ 차원 평가 |
| G7 | L2 | **대안**이 1개 이상 제시된다 (단일 답 아님) | "대안 A/B" 또는 "또 다른 접근" 문단 |
| G8 | L2 | **실행 가능성**이 구체적 (담당·기간·리소스) | 권고마다 "누가/언제/얼마나" 중 2개 이상 명시 |

## verifier가 하지 말 것 (편향 방지)

- 산출물을 **고쳐주지 않는다**. 판정만 한다. (improve는 원본 에이전트/사람이 한다)
- **expected.md 이외의 "이상적 답"을 추측해서 비교하지 않는다**. 시나리오 자체가 허용한 다양한 접근을 존중.
- 산출물 언어(한국어/영어)와 무관하게 **내용 기준**으로 판정. 스타일 트집은 평가 외.
